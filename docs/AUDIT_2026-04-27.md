### C-01 — Test script exposed in project root
SEVERITY: CRITICAL
FILE: `simulate-web-upload.php` (project root)
LINE: 1
ISSUE: A PHP test script that bootstraps the full Laravel application is committed to the project root. It can be executed via CLI or, if the document root is misconfigured, via HTTP.

### C-03 — `staff2_signature_data` has no size limit in UpdateStatusRequest
SEVERITY: CRITICAL
FILE: `app/Http/Requests/UpdateStatusRequest.php`
LINE: 41
ISSUE: The validation rule `'staff2_signature_data' => 'nullable|string'` imposes no maximum length. Applicant `signature_data` in StoreRequestRequest and UpdateRequestRequest is capped at `max:819200` (~800 KB string), but the staff2 approval signature has no cap.
IMPACT: A malicious or careless staff2 user can submit an arbitrarily large base64 string on every approval, causing memory exhaustion in PHP (base64 decoding), excessive database write size, and potential DoS of the transition endpoint.
FIX: Add `'max:819200'` to the `staff2_signature_data` validation rules in UpdateStatusRequest, mirroring the cap on applicant signatures.

### M-01 — RequestController::update() writes data outside WorkflowTransitionService transaction
SEVERITY: MEDIUM
FILE: `app/Http/Controllers/RequestController.php`
LINE: 212–247 (inner DB::transaction), 249 (executeTransition call)
ISSUE: When an admission user resubmits a RETURNED request, `update()` runs an inner `DB::transaction` that saves description, field_values, vot_items, and revision_count, then calls `WorkflowTransitionService::executeTransition()` OUTSIDE that transaction. ## Scope Compliance

**Does the system match the 4-role workflow spec?**
YES. The workflow enforces admission → staff1 → staff2 → (staff1 completes) with an audited staff2 override path. All four roles (admission, staff1, staff2, admin) are present and correctly scoped.

**Are all removed features (dean, deadline, priority) gone?**
PARTIALLY. The dean role is removed from transitions, routes, and the enum. However, dean-era schema columns, seeders, service code, and middleware remain. Deadline and priority columns persist in the database schema and are referenced by live code.

**Is the workflow enforced at every transition point?**
YES. Every status change routes through `WorkflowTransitionService::executeTransition()`. The transaction uses `lockForUpdate()` to prevent race conditions. Signature requirements, checklist requirements, and return/decline reasons are all enforced before transitions complete.

**Is the audit trail complete and append-only?**
MOSTLY. No update or delete calls on AuditLog were found. However, the initial submission audit log is written outside the DB::transaction (M-02), creating a small window for orphaned requests with no audit entry.

**Are there any features outside the defined scope?**
YES — several legacy features (priority, deadline, dean signature, advanced admin controllers, AdvancedRequestRequest) remain as dead code but do not affect the active workflow. They should be cleaned up to prevent confusion and accidental reactivation.
